# 1) Видение и идея проекта

Вы разрабатываете **веб-систему электронной выписки и отпуска рецептов** для аптечных сетей Казахстана, которая заменяет бумажные рецепты цифровым документом с жизненным циклом.

### Суть идеи

- **Рецепт становится цифровым объектом**, который создаёт врач и который может быть отпущен в аптеке.
- **QR-код — это ключ**, содержащий только **ID рецепта**, а все данные подтягиваются с сервера.
- Пациент **не устанавливает приложение** и **не регистрируется**: он получает **ссылку на QR** по SMS и email и показывает её в аптеке.
- Фармацевт сканирует QR, видит рецепт + данные пациента, **сверяет удостоверение личности**, и фиксирует отпуск (полный или частичный).
- QR **не одноразовый** и всегда показывает актуальный статус: **не выдан / частично / выдан полностью**, а также остатки по каждому препарату.

# 2) Участники и роли

Для дипломного MVP достаточно 3–4 роли:

1. **Doctor (Врач)**
- Создаёт электронный рецепт для пациента (ФИО, ИИН, телефон, email).
- Добавляет **несколько препаратов** в одном рецепте (дозировка, форма, количество, курс).
- Редактирует рецепт **по правилам** (см. раздел 6).
- Может отменить рецепт или отменить оставшуюся часть (см. раздел 7).
1. **Pharmacist (Фармацевт)**
- Сканирует QR, получает доступ к рецепту.
- Видит данные пациента, назначение и остатки.
- Проверяет личность по удостоверению личности.
- Отмечает отпуск по каждой позиции: **сколько выдал сейчас**.
- Фиксирует **частичный отпуск** и завершение рецепта.
1. **Admin (Админ аптечной сети/системы)**
- Управляет пользователями (врачи/фармацевты), ролями, доступами.
- Контролирует настройки системы, справочники (опционально), аудит.
- Может блокировать/отключать доступ, просматривать логи.
1. **Patient (Пациент)** — без аккаунта
- Получает ссылку на QR по SMS/email и показывает фармацевту.
- Не логинится и не вводит пароль (это сознательное упрощение для MVP).

# 3) Основные пользовательские сценарии (как “живёт” система)

## Сценарий A — Выписка рецепта врачом

1. Врач авторизуется.
2. Заполняет данные пациента: **ФИО + ИИН + телефон + email**.
3. Добавляет в рецепт **несколько препаратов**, у каждого:
    - название/МНН (как вы решите),
    - форма (таблетки/ампулы/сироп),
    - дозировка,
    - количество,
    - курс/инструкция.
4. Система создаёт рецепт со сроком действия **30 дней**.
5. Система генерирует QR, где хранится **только ID рецепта**.
6. Пациент получает ссылку на QR по **SMS и email** (в демо можно мок/заглушку).

## Сценарий B — Отпуск в аптеке по QR

1. Фармацевт авторизуется.
2. Сканирует QR через веб-камеру/сканер.
3. Система открывает рецепт по ID и показывает:
    - данные пациента (ФИО, ИИН),
    - список назначенных препаратов,
    - сколько назначено, сколько уже выдано, сколько осталось,
    - текущий статус рецепта.
4. Фармацевт сверяет личность по удостоверению личности.
5. Отмечает отпуск:
    - либо “выдать всё”,
    - либо по каждой позиции вводит “выдать сейчас X”.
6. Система обновляет остатки и статус рецепта.

## Сценарий C — Контроль врачом

- Врач видит статусы рецептов: активен/частично/полностью/отменён/истёк.
- Может редактировать/отменять по правилам ниже.

# 4) Состояния рецепта (state-машина)

Минимальные состояния:

1. **Created** — создан (можно объединить с Active)
2. **Active** — активен и доступен к отпуску
3. **Partially dispensed** — отпущен частично
4. **Dispensed** — отпущен полностью
5. **Cancelled** — отменён (или “закрыт”)
6. **Expired** — истёк срок (через 30 дней)

### Что показывает QR

При открытии ссылки/сканировании QR система отображает:

- статус рецепта (Active/Partially/Dispensed/Cancelled/Expired),
- по каждой позиции: назначено / выдано / осталось.

---

# 5) Частичный отпуск (как работает “остаток”)

Каждая позиция рецепта хранит:

- `qty_prescribed` — назначено врачом
- `qty_dispensed` — уже выдано аптекой
- `qty_remaining = qty_prescribed - qty_dispensed` — осталось выдать

Фармацевт при отпуске вводит “выдать сейчас” по каждой позиции, система:

- добавляет запись о выдаче (событие),
- увеличивает `qty_dispensed`,
- уменьшает `qty_remaining`,
- пересчитывает статус рецепта:
    - если по всем позициям `remaining=0` → **Dispensed**
    - иначе → **Partially dispensed** (если была хотя бы одна выдача)

# 6) Правила редактирования рецепта (включая после частичного отпуска)

Вы выбрали: **редактирование разрешено**, но нужно правило, чтобы не ломать историю выдачи.

## 6.1 До первой выдачи (статус Active)

Врач может редактировать рецепт свободно:

- менять список препаратов,
- менять дозировки, количество, курс,
- исправлять данные пациента (если нужно).

## 6.2 После частичного отпуска (статус Partially dispensed)

Редактирование становится **ограниченным**, чтобы сохранить корректность:

### Разрешено

1. **Добавить новую позицию** (новый препарат)
    - безопасно: у новой позиции `dispensed=0`.
2. **Увеличить назначенное количество** у позиции, по которой уже была выдача
    - безопасно: `prescribed` становится больше, остаток увеличивается.
3. **Уменьшить назначенное количество**, но только до значения **не меньше уже выданного**
    - правило: `new_qty_prescribed >= qty_dispensed`
    - иначе будет “выдано больше, чем назначено”, что нельзя.
4. **Изменять текстовые поля**, не влияющие на учет выдачи
    - комментарии, инструкции, примечания (если есть).

### Запрещено (после частичного отпуска)

1. Нельзя менять “сущность” уже отпускаемой позиции:
    - нельзя менять препарат/МНН,
    - нельзя менять дозировку/форму у позиции, где `qty_dispensed > 0`,
    - нельзя удалять позицию, если по ней была выдача.
2. Нельзя уменьшать назначенное меньше уже выданного.

### UX-логика (как это выглядит)

- Если по позиции была выдача (`dispensed>0`) — поля “препарат/форма/дозировка” становятся read-only.
- Доступны только правки по количеству (с ограничением) и добавление новых позиций.

---

# 7) Отмена рецепта (особенно после частичного отпуска)

Вы сказали “отмена по усмотрению врача”. Чтобы это было логично и не противоречило факту выдачи, делаем 2 режима отмены:

## 7.1 Отмена до любой выдачи (Active)

- Врач может **полностью отменить рецепт**.
- Статус становится **Cancelled**.
- Отпуск запрещён.

## 7.2 После частичного отпуска (Partially dispensed)

Полная отмена “задним числом” не имеет смысла, потому что часть уже выдана. Поэтому правильная модель:

### “Отмена остатка” (Cancel remaining / Close)

- Врач отменяет **оставшуюся невыданную часть** рецепта.
- Система фиксирует, что остаток аннулирован (не как “выдано”, а как “закрыто врачом”).
- Отпуск дальше запрещён.
- История сохраняется: сколько реально было выдано аптекой.

**Итоговый статус** можно выбрать как:

- **Cancelled** (и внутри пометить “cancelled remaining”), или
- отдельный “Closed”.

Для MVP достаточно: **Cancelled** + причина “Cancelled remaining after partial dispense”.

# 8) Архитектурное видение (минимальная архитектура системы)

## Frontend (Web)

1. **Портал врача**
    - создать рецепт
    - список рецептов и статусы
    - редактирование по правилам
    - отмена/отмена остатка
2. **Портал фармацевта**
    - скан QR
    - просмотр рецепта и пациента
    - ввод отпуска по позициям
    - подтверждение полного/частичного отпуска
3. **Публичная страница пациента**
    - открывается по ссылке из SMS/email
    - показывает QR-код (ID рецепта)
    - может показывать минимум инфо (например, “покажите в аптеке”)

## Backend (Node.js API)

- **Auth + RBAC** (Doctor/Pharmacist/Admin)
- **Prescription Service**
    - создание/чтение/редактирование/отмена
    - контроль срока 30 дней
- **Dispense Service**
    - выдача (частичная/полная)
    - пересчёт остатков и статуса
- **Notification Service**
    - отправка SMS/email (в демо — заглушка)
- **Audit Log**
    - кто/когда/что изменил или выдал

## Database

- `Users`
- `Patients` (ФИО, ИИН, телефон, email)
- `Prescriptions` (статус, даты, врач)
- `PrescriptionItems` (препарат, дозировка, qty_prescribed, qty_dispensed)
- `DispenseEvents` (кто выдал, когда, сколько, по каким позициям)
- `AuditLogs`
